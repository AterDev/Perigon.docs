# 官方模块

Perigon 官方模块由 [Perigon.Modules](https://github.com/AterDev/Perigon.Modules) 仓库维护。创建解决方案时可以选择官方模块；也可以在创建后查看清单并按需安装：

```pwsh
perigon module list
perigon module install Perigon.SystemMod AdminService
```

最新模板不会默认加入这些模块。每个模块都是以 `Mod` 结尾的独立程序集，业务实现和 DTO 位于 `src/Modules/{Name}Mod`，实体位于 `src/Definition/Entity/{Name}Mod`。

## 官方模块目录

| 模块 | 适用场景 | 主要能力 |
| --- | --- | --- |
| `Perigon.SystemMod` | 后台管理和身份权限基础 | 管理员用户、角色、权限、菜单、组织、系统配置与日志；包含初始化和异步日志处理。 |
| `Perigon.CMSMod` | 内容管理 | 文章与文章分类的后台管理能力。 |
| `Perigon.ResourceMod` | 面向租户的通用资源管理 | 管理资源环境、分类、分组、标签、资源定义及动态属性，并按角色、环境和分类配置资源访问权限。 |

### SystemMod

`SystemMod` 提供后台管理的基础领域模型，包括用户、角色、权限、权限组、菜单、组织和系统配置。它还提供系统日志记录与后台处理能力。需要管理端登录、角色授权或其他官方模块依赖角色能力时，通常应先安装该模块。

```pwsh
perigon module install Perigon.SystemMod AdminService
```

### CMSMod

`CMSMod` 用于基础内容管理，包含文章分类和文章。适合需要在管理端维护文章、公告或其他结构化内容的项目。

```pwsh
perigon module install Perigon.CMSMod AdminService
```

### ResourceMod

`ResourceMod` 是新增的通用资源管理模块，面向租户管理可分类、可配置属性的资源。资源创建依赖环境、分类和资源定义，可选分组与标签；资源定义决定动态属性的名称、类型、必填和长度约束。

模块还提供按“角色 + 环境 + 分类”的资源权限配置：管理员可维护资源与配置，非管理员只能读取服务端根据授权返回的资源。它适合管理内部服务、资产、链接、运行环境或其他需要分类与权限控制的资源目录。

```pwsh
perigon module install Perigon.ResourceMod AdminService
```

## 包内容与前端

官方模块包始终包含模块后端所需的实体、模块程序集、控制器和元数据。打包时若提供 `--front-path`，还会包含该前端模块目录及同级 `share` 目录，布局为：

```text
Frontend/{module}/...
Frontend/share/...
```

安装带前端内容的包时，将 `--front-path` 指向目标前端的 `modules` 根目录；模块与 `share` 分别恢复到该目录下。模块文件会覆盖同名文件，而 `share` 中已有的同名文件会保留。

模块包不包含前端应用外壳、根依赖配置或 npm/pnpm 依赖，也不会修改目标服务的 `Program.cs`。前端打包的完整用法和限制见[命令行文档](../代码生成/命令行.md)。

## CommonMod

`CommonMod` 是保留模块，用于承载多个模块可复用的基础功能；它不是官方模块清单中的可选业务模块。
